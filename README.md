# Информационная система для записи пациентов в поликлинику

## Описание проекта


Данный проект представляет собой кроссплатформенную информационную систему для автоматизации записи пациентов на прием в поликлинику, хранения справочной информации о врачах и услугах, а также ведения медицинских записей по оформленным приемам.

Изначально проект был реализован как WPF-приложение под Windows, однако в текущей версии был переработан в веб-приложение, которое может запускаться на разных операционных системах, включая macOS, Windows и Linux. Это позволяет использовать систему через браузер без привязки к конкретной платформе.

Проект ориентирован на демонстрацию основных принципов проектирования базы данных, построения клиент-серверного взаимодействия и реализации типовых операций медицинской регистрационной системы.

## Тема курсового проекта

Тема проекта:

`Разработка базы данных и информационной системы для регистрации пациентов в поликлинике`

## Цель проекта

Целью проекта является разработка информационной системы, которая позволяет:

- хранить данные о пациентах;
- хранить данные о поликлиниках;
- хранить данные о врачах и их специализациях;
- хранить перечень медицинских услуг;
- формировать доступные слоты записи к врачу;
- записывать пациента на прием;
- искать существующие записи;
- вести медицинские записи по уже созданным приемам.

## Описание базы данных

### Сущности

В проекте реализована нормализованная модель базы данных, включающая 9 сущностей:

1. `patient` — хранит данные о пациентах.
2. `policlinic` — хранит данные о поликлиниках.
3. `specialization` — хранит справочник медицинских специализаций.
4. `doctor` — хранит данные о врачах.
5. `service` — хранит список медицинских услуг.
6. `doctor_service` — связывает врача с услугой и стоимостью.
7. `doctor_availability` — хранит доступные слоты записи к врачу.
8. `appointment` — хранит записи пациентов на прием.
9. `medical_record` — хранит медицинские записи по приемам.

### Причины выбора такой структуры

Выделение отдельных сущностей необходимо для устранения избыточности и для корректного отражения предметной области.

Например:

- одна поликлиника может содержать много врачей;
- один врач относится к одной специализации;
- один врач может оказывать несколько услуг;
- одна услуга может оказываться разными врачами;
- один пациент может иметь несколько записей на прием;
- один прием может иметь одну медицинскую запись.

### Почему поликлиника вынесена в отдельную сущность

Сущность `policlinic` выделена отдельно, потому что:

- врач работает в конкретной поликлинике;
- одна поликлиника связана с несколькими врачами;
- пациент выбирает не только врача, но и медицинскую организацию;
- данные о поликлинике не должны дублироваться в каждой записи врача;
- это позволяет хранить адрес и телефон учреждения централизованно.

## Связи между сущностями

В проекте реализованы следующие основные связи:

- `policlinic 1:M doctor`
- `specialization 1:M doctor`
- `doctor 1:M doctor_availability`
- `doctor M:N service` через таблицу `doctor_service`
- `patient 1:M appointment`
- `doctor_service 1:M appointment`
- `appointment 1:1 medical_record`

## Нормализация базы данных

База данных приведена к нормализованному виду.

Основные признаки нормализации:

- данные о пациентах, врачах, поликлиниках и услугах разделены по отдельным таблицам;
- исключено лишнее дублирование строковых данных;
- связи многие-ко-многим вынесены в отдельную таблицу;
- медицинская запись хранится отдельно от самой записи на прием;
- справочники вынесены в отдельные сущности.

## Функциональные возможности системы

Текущая версия веб-приложения поддерживает следующие функции.

### 1. Регистрация пациента

Пользователь может:

- ввести ФИО пациента;
- указать, есть ли отчество;
- указать возраст;
- указать адрес;
- указать телефон;
- указать дату рождения;
- сохранить пациента в базе данных.

### 2. Оформление записи на прием

Пользователь может:

- выбрать пациента из базы;
- выбрать поликлинику;
- выбрать специализацию;
- выбрать врача;
- выбрать услугу;
- выбрать доступный слот;
- оформить запись на прием.

### 3. Поиск записи

Пользователь может:

- выбрать пациента;
- выбрать услугу;
- получить найденную запись;
- посмотреть статус записи;
- увидеть врача, поликлинику и дату приема.

### 4. Просмотр всех записей

Пользователь может:

- открыть список всех текущих записей;
- посмотреть пациента, врача, услугу, дату и статус;
- выбрать нужную запись для дальнейшей работы.

### 5. Ведение медицинской записи

Для выбранного приема можно:

- указать диагноз;
- указать лечение;
- добавить примечания;
- сохранить медицинскую запись в таблицу `medical_record`.

### 6. Администрирование справочников

Можно добавить:

- новую поликлинику;
- нового врача;
- специализацию;
- услугу;
- стоимость услуги;
- доступную дату и время приема.

## Валидация данных

В системе реализована валидация пользовательского ввода.

Проверяются:

- ФИО пациента;
- ФИО врача;
- наличие или отсутствие отчества;
- обязательность заполнения полей;
- возраст;
- телефон;
- стоимость услуги;
- дата рождения;
- будущая дата доступности;
- выбор сущностей из справочников.

### Примеры проверок

- если ФИО заполнено не полностью, система выводит сообщение о необходимости заполнить поле до конца;
- если отчество отсутствует, пользователь может поставить галочку `Нет отчества`;
- телефон должен содержать корректное количество цифр;
- возраст должен быть целым числом в допустимом диапазоне;
- дата рождения не может быть в будущем;
- дата доступности врача должна быть будущей;
- нельзя оформить запись без выбора врача, услуги и времени.

## API веб-приложения

Серверная часть предоставляет набор HTTP API.

Основные маршруты:

- `GET /api/health` — проверка работоспособности сервера;
- `GET /api/dashboard` — статистика по системе;
- `GET /api/patients` — список пациентов;
- `GET /api/policlinics` — список поликлиник;
- `GET /api/specializations` — список специализаций;
- `GET /api/doctors` — список врачей;
- `GET /api/services` — список услуг;
- `GET /api/availability` — доступные слоты врача;
- `GET /api/appointments` — список всех записей;
- `GET /api/appointments/search` — поиск записи;
- `GET /api/medical-records` — получение медзаписи по приему;
- `POST /api/patients` — добавление пациента;
- `POST /api/admin/doctor` — добавление врача, услуги и слота;
- `POST /api/appointments` — создание записи на прием;
- `POST /api/medical-records` — создание или обновление медзаписи.

## Инструкция по запуску

### Что нужно для запуска

Перед запуском проекта на компьютере должны быть установлены:

- `git`
- `node.js`
- `npm`
- `PostgreSQL`
- утилита `psql`

### Команды для запуска в терминале

Ниже приведена последовательность команд, которые нужно ввести в терминале после клонирования репозитория:

```bash
git clone <ссылка-на-репозиторий>
cd kursovaya/web_app
npm run init-db
npm start
```

После запуска нужно открыть в браузере:

```text
http://127.0.0.1:3000
```

### Что делают команды

Команда:

```bash
npm run init-db
```

выполняет:

- создание базы данных `hosdb`, если она отсутствует;
- применение схемы базы данных из `database_schema.sql`;
- загрузку тестовых данных из `web_app/db/seed.sql`.

Команда:

```bash
npm start
```

выполняет:

- запуск локального Node.js сервера;
- подключение к базе данных PostgreSQL;
- публикацию веб-интерфейса для работы через браузер.
